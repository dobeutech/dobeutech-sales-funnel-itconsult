## 2024-03-09 - @neondatabase/serverless Parameterized Queries Change
**Vulnerability:** The codebase is using `sql(query, params)` with `@neondatabase/serverless` which is no longer supported and throws a runtime error.
**Learning:** In newer versions of `@neondatabase/serverless`, calling `sql(query, params)` throws a TypeError: `This function can now be called only as a tagged-template function: sql\`SELECT \${value}\`, not sql("SELECT $1", [value], options). For a conventional function call with value placeholders ($1, $2, etc.), use sql.query("SELECT $1", [value], options).`
**Prevention:** Always use `sql\`SELECT * FROM table WHERE col = \${val}\`` or `sql(query, params)` -> `sql(query, params)` is wrong, must use tagged templates or if dynamically building, `sql(query, params)` doesn't exist, we must use tagged templates `sql\`SELECT * FROM prospects WHERE status = \${status} ORDER BY overall_score DESC\``.

## 2024-05-06 - Environment Enumeration in Health Endpoints
**Vulnerability:** Public-facing health endpoints (`src/app/api/health/route.ts`) explicitly checking for the presence of sensitive environment variables (e.g., `COMPOSIO_API_KEY`) and returning a binary status ('ok' or 'missing').
**Learning:** Returning binary checks for environment variables allows malicious actors to map the backend environment and identify what third-party services are configured. This is a form of environment enumeration and information disclosure.
**Prevention:** Health checks should only verify that the application has successfully instantiated critical services or SDKs internally, without directly exposing the configuration state of individual environment variables to the public. If an SDK fails to initialize due to missing keys, it should be reflected as a degraded service status rather than explicitly stating the missing key.
