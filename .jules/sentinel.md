## 2024-03-09 - @neondatabase/serverless Parameterized Queries Change
**Vulnerability:** The codebase is using `sql(query, params)` with `@neondatabase/serverless` which is no longer supported and throws a runtime error.
**Learning:** In newer versions of `@neondatabase/serverless`, calling `sql(query, params)` throws a TypeError: `This function can now be called only as a tagged-template function: sql\`SELECT \${value}\`, not sql("SELECT $1", [value], options). For a conventional function call with value placeholders ($1, $2, etc.), use sql.query("SELECT $1", [value], options).`
**Prevention:** Always use `sql\`SELECT * FROM table WHERE col = \${val}\`` or `sql(query, params)` -> `sql(query, params)` is wrong, must use tagged templates or if dynamically building, `sql(query, params)` doesn't exist, we must use tagged templates `sql\`SELECT * FROM prospects WHERE status = \${status} ORDER BY overall_score DESC\``.

## 2024-05-01 - Prevent leaking database credentials in Vite frontend
**Vulnerability:** The Vite frontend application (`my-neon-app`) was initializing a database connection directly using `import.meta.env.VITE_DATABASE_URL` and `@neondatabase/serverless`. Variables prefixed with `VITE_` are injected into the client bundle, exposing sensitive database credentials to users.
**Learning:** Any environment variables accessed via `import.meta.env` with a `VITE_` prefix are fully visible in the compiled client-side JavaScript. This allows malicious actors to directly connect to the database and potentially steal, manipulate, or delete data.
**Prevention:** Always ensure that database connections and other secure credentials reside solely in server-side code (e.g., Next.js API routes). Ensure the frontend only communicates with a Backend-for-Frontend (BFF) via standard API calls (`fetch`), rather than making direct database queries from the client. Never prefix database strings or sensitive API keys with `VITE_` or `NEXT_PUBLIC_`.
