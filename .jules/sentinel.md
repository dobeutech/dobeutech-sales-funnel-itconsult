## 2024-03-09 - @neondatabase/serverless Parameterized Queries Change
**Vulnerability:** The codebase is using `sql(query, params)` with `@neondatabase/serverless` which is no longer supported and throws a runtime error.
**Learning:** In newer versions of `@neondatabase/serverless`, calling `sql(query, params)` throws a TypeError: `This function can now be called only as a tagged-template function: sql\`SELECT \${value}\`, not sql("SELECT $1", [value], options). For a conventional function call with value placeholders ($1, $2, etc.), use sql.query("SELECT $1", [value], options).`
**Prevention:** Always use `sql\`SELECT * FROM table WHERE col = \${val}\`` or `sql(query, params)` -> `sql(query, params)` is wrong, must use tagged templates or if dynamically building, `sql(query, params)` doesn't exist, we must use tagged templates `sql\`SELECT * FROM prospects WHERE status = \${status} ORDER BY overall_score DESC\``.

## 2024-05-11 - Exposed Database URL in Vite Frontend Client
**Vulnerability:** The VITE_DATABASE_URL environment variable and @neondatabase/serverless client were used directly in the Vite client-side bundle (my-neon-app/src/lib/db.ts), exposing the database connection string and allowing direct unauthorized database execution.
**Learning:** Vite bundles any environment variables prefixed with `VITE_` into the static frontend payload. The client application should never connect to a database directly; it must route interactions through an authenticated backend API layer to maintain a secure boundary and prevent secret leakage.
**Prevention:** Strictly prohibit `VITE_` or `NEXT_PUBLIC_` prefixes on sensitive environment variables (such as connection strings or API keys). Use a Backend-For-Frontend (BFF) architecture using tools like Next.js App Router for database interactions, and enforce frontend communications to use HTTP calls (e.g. `fetch`) via a proxy configuration (`vite.config.ts`).
